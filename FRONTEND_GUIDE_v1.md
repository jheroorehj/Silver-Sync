## 2. UI/UX 필수 규정 — 모든 컴포넌트에 적용

의료법·개인정보 보호법·EMR 인증 기준·KRDS 가이드라인 기반 규정입니다. 새 컴포넌트를 만들거나 기존 컴포넌트를 수정할 때 반드시 준수하세요.

### 2.1 접근성 & 가독성

- 본문 최소 `16px`, 산세리프체
- 텍스트-배경 명도 대비 **4.5:1 이상** (WCAG AA)
- 버튼·인터랙티브 요소 최소 `44×44px`, 요소 간 간격 `≈8px` 이상
- 아이콘 단독 사용 금지 → 텍스트 라벨 병기 필수

### 2.2 보안 UI 필수 항목

| 항목 | 구현 방법 |
|------|-----------|
| 마스킹 | 주민번호·환자 식별번호 → `900101-1******` 형태 마스킹 |
| 세션 타임아웃 | 10~30분 미사용 시 경고 모달 → 자동 로그아웃 |
| 접근 로그 | 환자 정보 조회·수정·삭제 시 로그 생성 API 호출 트리거 |
| 수정 이력 UI | EMR 데이터 수정 시 수정 전/후 diff 뷰 + 수정 사유 입력 필드 |

### 2.3 FHIR / KR Core 데이터 바인딩

- API 응답은 KR Core FHIR 리소스(Patient, Condition, Observation 등) 기반
- FHIR → 내부 ViewModel 변환 어댑터 레이어를 별도 분리
- 날짜·코드 시스템(SNOMED, KCD 등)은 한글 라벨로 매핑

### 2.4 임상 안전 규칙

- 수치 표시 시 **단위(unit)와 참고 범위(referenceRange)** 필수 병기
- AI 생성 콘텐츠에 면책 배너 고정: `"본 내용은 AI 에이전트가 생성한 참고 자료이며, 최종 임상 판단은 담당 의료진의 책임입니다."`
- triageLevel `emergent` 시 시각적·텍스트 경고 강화
- severity를 색상에만 의존하지 말고 아이콘+텍스트 라벨 병기

## 3. 향후 개선 방향

> **"무엇을 해야 하는가"만 정의합니다. "어떻게·어디에" 구현할지는 기존 소스 트리를 직접 읽고 판단하세요.**

### 3.1 당뇨 연속 데이터(CGM) 시계열 표시

- 백엔드에서 CGM 등 시계열 바이탈 데이터가 내려올 예정
- SOAP 노트 Objective 섹션 내에 시계열 차트(라인/에어리어)를 삽입할 수 있도록 확장 고려
- 이상 구간(referenceRange 이탈) 하이라이트, 시간축 줌/팬 인터랙션 검토
- 기존 VitalSign 타입에 시계열 배열을 수용할 수 있는 확장 필요

### 3.2 보안 UI 강화

- 세션 타임아웃 로직이 아직 없다면 추가
- MaskedId 처리가 모든 환자 식별번호 노출 지점에 적용되었는지 점검
- 접근 로그 API 호출 트리거가 실제 백엔드 엔드포인트에 연결 가능한 구조인지 확인

### 3.3 FHIR 어댑터 레이어 분리

- 현재 patients.json 목업 데이터를 직접 사용 중이므로, 실제 FHIR API 전환 시 변환 로직이 한 곳에 모이도록 어댑터 분리 검토
- KR Core 프로파일 필드 매핑 누락 여부 확인

### 3.4 인쇄/PDF 내보내기 품질

- @media print 스타일이 SOAP 노트 4-섹션에서 페이지 브레이크가 자연스러운지 검증
- PDF 내보내기 시 차트·게이지 등 인터랙티브 요소가 정적 이미지로 올바르게 렌더링되는지 확인

### 3.5 에이전트 메타데이터 투명성

- 에이전트 판단 근거(confidenceScore, dissensionFlag, debateLog)가 의사 뷰에 노출되는지 확인
- 없다면 SOAP 노트 하단에 접기/펼치기 패널로 추가
- `dissensionFlag === true` 시 "⚠ 에이전트 의견 불일치" 경고 표시

### 3.6 접근성 검수

- 기존 구현 컴포넌트 일괄 점검:
  - aria-label, aria-expanded, role 속성 적절 사용 여부
  - 색상 단독 정보 전달 없이 아이콘+텍스트 병기 여부
  - 키보드 내비게이션 (접기/펼치기, 탭 전환 등)

## 5. 에이전트 작업 규칙

1. **소스 트리를 먼저 읽어라**: 태스크 시작 시 프로젝트 디렉토리를 탐색하고, 기존 파일 구조·네이밍·패턴을 파악한 뒤 계획을 세울 것.
2. **디렉토리 구조를 임의로 변경하지 마라**: 기존 구조를 따르되, 변경이 필요하면 이유를 명시하고 사용자 승인 후 진행.
3. **기존 공통 컴포넌트를 재사용하라**: NumberSpinner, StatusBadge 등이 이미 존재하면 새로 만들지 말고 재사용하거나 확장.
4. **타입을 기존 타입 파일에 추가하라**: 새 인터페이스가 필요하면 기존 타입 파일 위치를 찾아서 거기에 추가.
5. **목업 데이터와 실제 API를 분리하라**: patients.json 같은 목업은 어댑터 레이어 뒤에 두어, API 전환 시 컴포넌트 코드 변경 없이 교체 가능하게 유지.
